# FeiKnow（飞知）智能知识助手 — 项目总结文档

---

## 一、项目概述

**FeiKnow（飞知）** 是一款面向研发团队的「主动式」CLI 终端知识调度引擎，联动飞书生态，实现知识获取范式从 **"人找知识"** 到 **"知识找人"** 的跃迁。

| | 说明 |
|---|---|
| 产品名称 | FeiKnow（飞知） |
| 产品定位 | 极客专属的终端知识调度引擎 |
| 目标用户 | 后端开发工程师、DevOps 运维人员、算法研究员 |
| 核心场景 | CLI 终端报错自动捕获 → RAG 私域知识检索 → 终端 + 飞书群双通道推送修复建议 |
| 技术栈 | Spring Boot 3.3 + PostgreSQL/pgvector 向量库 + DeepSeek Chat API + 飞书 IM/Webhook |

## 二、维度 1：完整性与价值（50%）

### 2.1 解决什么痛点？

当开发者在终端运行代码、编译框架或启动微服务时，常会遇到各种报错。传统解决路径：

> 复制报错 → 切换浏览器 → 打开内部 Wiki → 搜索关键词 → 在历史记录中筛选方案

这一过程**严重打断开发者的"心流"**，且极度依赖个人检索能力和对知识库位置的记忆。

### 2.2 AI 起到什么关键作用？

| AI 能力 | 作用 |
|----------|------|
| **关键词向量化** | 将报错文本转为 1024 维技术特征向量（覆盖 100+ 研发关键词：Docker、K8s、Maven、Nacos 等） |
| **混合检索（语义 + 关键词）** | `0.7 × 向量余弦相似度 + 0.3 × 关键词命中率`，弥补纯语义检索对代码报错匹配不足的缺陷 |
| **RAG 增强生成** | 将 Top-3 知识片段注入 System Prompt，约束 DeepSeek 仅根据内部文档回答，杜绝幻觉 |
| **智能降级策略** | 相似度 < 0.6 时自动切换通用大模型，并显式标注 `[来自通用大模型]` |

### 2.3 流程是否完整闭环？

## 三、维度 2：创新性（25%）

### 3.1 AI 相关创新点

#### 创新 1：终端原位救援模式

通用 AI 产品都局限在 Web/IM 对话框，开发者必须"离开案发现场"去对话。FeiKnow 的 **Python CLI 作为命令代理**（`feiknow <command>`），自动捕获异常并上报，将 AI 能力无缝嵌入开发者最熟悉的工作环境——终端。

#### 创新 2：双栖分发策略

| 形态 | 通道 | 特点 |
|------|------|------|
| 本地极客态 | CLI 终端 | 精简修复指令，直接可复制执行 |
| 团队协作态 | 飞书交互式卡片 | 报错摘要 + AI 建议 + 参考文档链接 |

#### 创新 3：语义 + 关键词混合检索

针对代码报错的特殊性（堆栈信息高度结构化、关键词密集），独创 `(0.7 × 余弦相似度) + (0.3 × PostgreSQL ~* 正则命中)` 的混合排序算法，在 pgvector 层面用一条 SQL 完成。

#### 创新 4：关键词特征向量引擎

`AiProviderClient` 内置 100+ 研发技术关键词（覆盖 Docker/K8s/Spring/Maven/Nacos/Redis 等），将报错文本映射为 1024 维**二进制特征向量**（含关键词命中位 + L2 归一化），在保证检索精度的同时无需依赖外部 Embedding API（省钱且不依赖第三方）。

### 3.2 方案差异化亮点

| 对比维度 | ChatGPT/通用 AI | 传统 Wiki 搜索 | **FeiKnow** |
|----------|:---:|:---:|:---:|
| 工作场景 | Web 对话框 | 浏览器页面 | **终端原位** |
| 知识来源 | 公域训练数据 | 人工维护 | **私域向量库 + RAG** |
| 触发方式 | 手动输入 | 手动搜索 | **事件驱动自动触发** |
| 团队协作 | 无 | 无 | **飞书卡片广播 + 来源链接** |
| 可信度标注 | 无来源 | 有原文 | **明确标注来源 + 降级说明** |

### 3.3 可复用性与可推广性

- **架构解耦**：`AiProviderClient` 独立管理 AI 调用，飞书 IM 通过 `FeishuApiClient` 独立管理，替换任一组件无需改动业务代码
- **跨平台 CLI**：Python 3 客户端不依赖特定 shell，Windows/Linux/macOS 均可
- **知识源可扩展**：当前监听飞书文档 Webhook，可扩展 Confluence/语雀/GitLab Wiki
- **场景可推广**：同架构可用于代码 Review 助手、部署失败诊断、API 异常排查等

## 五、核心部分代码展示

### 5.1 混合检索 RAG 核心引擎

```java
// RescueService.java — RAG 核心逻辑
public String rescue(String errorText) {
    // 1. 报错文本 → 关键词向量
    float[] errorEmbedding = aiProviderClient.createEmbedding(errorText);
    // 2. 提取关键词 Pattern（停用词过滤）
    String keywordPattern = extractKeywordPattern(errorText);
    // 3. 混合检索（语义 70% + 关键词 30%）
    List<Object[]> similarChunks = knowledgeChunkRepository
        .findTop3Hybrid(errorEmbedding, keywordPattern);
    
    // 4. 相似度判断 → RAG 或降级
    if (maxSimilarity >= 0.6) {
        // 私域 RAG：注入知识 Context
        messages.add(Map.of("role","system", SYSTEM_PROMPT + "\nContext:\n" + context));
        result = aiProviderClient.createChatCompletion(messages);
    } else {
        // 降级：通用大模型，显式标注
        result = "[来自通用大模型]\n" + aiProviderClient.createChatCompletion(messages);
    }
    // 5. 飞书卡片异步推送
    feishuCardService.sendErrorNotification(errorText, result, sourceUrls);
}
```

### 5.2 混合检索 SQL（pgvector）

```sql
-- KnowledgeChunkRepository.java - 一条 SQL 完成混合排序
SELECT id, content, source_url, embedding,
    (0.7 * (1 - (embedding <=> CAST(:embedding AS vector))) +
     0.3 * CASE WHEN content ~* :keywordPattern THEN 1.0 ELSE 0.0 END
    ) as similarity
FROM knowledge_chunk
ORDER BY similarity DESC
LIMIT 3
```

## 六、AI 亮点介绍

### 6.1 项目中使用了哪些高阶 AI 技巧？

| 技巧 | 说明 |
|------|------|
| **混合检索 (Hybrid Search)** | 语义向量 + 关键词正则，独创 7:3 权重，弥补纯语义对代码报错匹配不足 |
| **RAG 阈值降级** | 相似度不足时光滑降级至通用大模型，且显式标注来源 |
| **关键词向量引擎** | 自研 100+ 研发技术关键词二进制向量，L2 归一化，零外部 API 依赖 |
| **System Prompt 约束** | `"仅根据提供的Context回答，如果Context不包含解决方案请直说不知道"`，严防幻觉 |
| **滑动窗口文本分块** | 500 字符 + 50 重叠，避免语义在分块边界处断裂 |

### 6.2 人和 AI 的分工

| 人的职责 | AI 的职责 |
|----------|----------|
| 编写《踩坑指南》等文档沉淀经验 | 自动监听文档更新，分块入库 |
| 在终端正常运行命令 | 自动捕获失败命令的 stderr |
| 判断修复建议是否采纳 | 从私域知识库检索 + 生成修复建议 |
| 管理团队知识库 | 在飞书群自动广播 Debug 卡片 |

### 6.3 核心模型选型思路

1. **放弃飞书 AI API**：经 15 种路径测试确认 `/open-apis/ai/v1/` 已不对外公开
2. **Embedding 选型**：DeepSeek 无公开 Embedding API → 自研关键词向量引擎（效果足够且免费）
3. **Chat 选型**：DeepSeek `deepseek-chat` 模型——成本极低、中文友好、兼容 OpenAI 格式
4. **架构预留**：`AiProviderClient` 支持环境变量配置，随时可切换为 OpenAI/智谱/MiniMax

### 6.4 AI 引入后对原有工作流的改变

| 原来 | 现在 |
|------|------|
| 报错 → 手动搜索 Wiki | 报错 → FeiKnow 自动分析 → 终端直接打印建议 |
| 知识分散在个人笔记 | 文档更新 → 自动向量化入库沉淀 |
| 踩坑经验个人独享 | Debug 卡片自动广播到团队飞书群 |
| 新人遇到问题反复问 | 知识库自动匹配历史方案 |

| 混合检索 SQL | `KnowledgeChunkRepository.java` | pgvector `<=>` + PostgreSQL `~*` 正则加权排序 |
| CLI 终端代理 | `feiknow.py` | Python 命令包装器，自动捕获 stderr 并上报 |

**亮点 Demo**：
- `feiknow.py` 执行任意命令 → 失败时自动上报 → 后端混合检索 → DeepSeek 生成修复建议 → 返回终端
- 混合检索效果验证：GitLab OOM / 微服务 503 / Java 堆溢出 → 均成功命中私域知识库
