# 《FeiKnow 智能知识助手 - 场景定义文档》

## 1. 项目定位与核心愿景
* **产品名称：** FeiKnow（飞知）—— 极客专属的终端知识调度引擎
* **产品定位：** 一款驻留在开发者 CLI 终端，联动飞书生态的“主动式”场景化知识助手。
* **核心愿景：** 打破企业内部信息孤岛，实现知识获取范式从“人找知识（被动检索）”向“知识找人（主动救援）”的跃迁。

## 2. 目标用户与痛点分析
* **目标用户群：** 研发团队成员，包括后端开发工程师、算法研究员及 DevOps 运维人员。
* **核心痛点（Context Switching）：**
  当开发者在终端运行代码、编译框架或启动微服务时，常会遇到各种报错（如容器启动失败、特定框架依赖冲突）。传统的解决路径是：**复制报错信息 -> 切换到浏览器 -> 打开内部 Wiki / 飞书文档 -> 搜索关键字 -> 在海量历史记录中筛选排查方案**。
  这种“被动搜索”模式严重打断了开发者的“心流（Flow）”，且极度依赖开发者个人的检索能力和对知识库所在位置的记忆。

## 3. 为什么“通用大模型搜索”无法满足？
虽然市面上已有众多强大的通用对话大模型，但它们在企业研发场景中存在致命盲区：
1. **缺乏私域上下文：** 通用模型无法知道企业内部特殊的微服务网关配置、历史发生过的同类 Bug 处理记录，或是特定业务线（如高并发秒杀系统、特定硬件训练框架）的独有踩坑指南。
2. **缺乏场景穿透力：** 通用模型通常局限在“Web 对话框”中。开发者必须离开终端去对话，而 FeiKnow 的设计理念是将知识直接“输送”到报错发生的“案发现场（终端）”。

## 4. 核心业务场景：CLI 终端错误的主动救援 (Event-Driven Rescue)
FeiKnow 将围绕“终端报错”这一高频、高痛点场景展开闭环服务：

* **被动知识吸收（Data Ingestion）：**
  后台服务通过飞书 Webhook 持续监听团队内部的《研发踩坑指南》、《系统部署架构文档》、《历史 Incident 复盘》等文档与会议纪要。一旦有更新，立刻通过 OpenClaw 进行清洗、分块并向量化入库，构建高密度的私域知识图谱。
* **事件驱动触发（Trigger）：**
  开发者在终端配置 FeiKnow CLI 工具。当执行特定脚本失败并抛出异常堆栈（Exception Stack Trace）时，CLI 会自动捕获报错上下文，并向 FeiKnow 后端发起求助。
* **双栖主动分发（Active Delivery）：**
  * **形态 1（本地极客态）：** 后端利用 RAG 技术匹配内部知识后，直接在终端界面（CLI）打印精简的修复指令或配置项修改建议，开发者甚至可以直接复制执行。
  * **形态 2（团队协作态）：** FeiKnow 同步调用飞书 API，在关联的“研发支持群”或“项目群”中推送一张**「Debug 协助卡片」**。卡片包含：报错摘要、可能的原因分析、内部文档引用链接以及历史处理过该问题的负责人艾特（@），快速拉齐团队信息差。

## 5. 效果验证与自证指标 (Prove it)
为了向评委证明 FeiKnow 不是一个产生“幻觉”的玩具，我们将从以下三个维度进行数据对账与验证：

1. **内容准确性（Accuracy & Grounding）：**
   * **指标：** 飞书卡片中附带明确的 `[引用来源：飞书文档 - XXXX]` 标签的比例达到 100%。
   * **自证：** 通过严格的 RAG prompt 限制，确保所有生成的排查建议必须强关联内部已有的文档库，坚决杜绝模型自行编造的无效配置参数。
2. **用户接受度（Acceptance）：**
   * **指标：** 统计飞书推送卡片中“查看原文档”或“复制解决指令”按钮的点击率（CTR）。
   * **自证：** 部署简单的埋点，展示一周内开发者采纳 FeiKnow 建议的次数，以及卡片下方互动按钮（👍 解决 / 👎 未解决）的反馈数据。
3. **流转效率提升（Velocity）：**
   * **指标：** 故障平均排查时间（MTTR, Mean Time To Repair）的缩短。
   * **自证：** 模拟对比测试——针对一个内部特有的配置错误，人工去飞书搜索耗时（约 3-5 分钟），对比 FeiKnow 终端毫秒级自动推送的时间（< 5 秒），直观展示效率的成倍提升。
